
# 商业级起名系统 — 补充规范文档

## 一、评分算法详细定义

### 1.1 概述

1.md 给出了评分维度与权重，但缺少具体计算公式。本节补齐每个维度的量化方法、归一化方式和聚合函数。

### 1.2 各维度评分函数

#### 1.2.1 字义正向性（权重 20%）

```
function scoreMeaning(char):
  正向词库包含约 3000 个积极义字（吉、祥、瑞、慧、轩…）
  中性字默认 0.5，积极字根据等级映射到 0.6~1.0
  消极字返回 0

  名字得分 = 各字 mean(scoreMeaning(c)) × 0.8
            + 组合义正向性 × 0.2
  组合义正向性：查组合义词典（如有），否则用各字均值
```

**数据来源**：由 2.md 的 `meaning` 字段结合人工标注正向等级（1~5 档）。  

**归一化**：天然 0~1。

#### 1.2.2 音律顺滑度（权重 20%）

```
function scoreTone(name):
  音调序列 = toTones(pinyin(name)) // 如 [1, 2, 4]
  声母序列 = toInitials(pinyin(name))

  规则：
    a) 相邻字声调变化数 ≥ 2 加分（避免同调堆叠）
    b) 声母重复扣分（如 lǐ lì → 声母重复）
    c) 韵母重复扣分（如 ān ān → 韵母重复）
    d) 四声搭配最佳模式：平仄相间 + 末字为平声加分

  评分 = baseScore(0.7)
         + 0.15 × toneVariationScore
         - 0.10 × repetitionPenalty
         + 0.05 × endingToneBonus

  归一化到 0~1（通过 clamp 或 sigmoid）
```

**多音字处理**：取该字在姓名场景中的首选读音（由 2.md 频率统计决定），若首选读音不可用则取次选。

#### 1.2.3 风格匹配度（权重 15%）

```
function scoreStyle(name, targetStyle):
  名字风格标签 = union(各字 vibe 标签) + 组合词 vibe 标签
  匹配度 = |名字风格标签 ∩ targetStyle| / |targetStyle|

  归一化：天然 0~1
```

若未指定风格，所有风格各算一次，取最高分。

#### 1.2.4 常用度/可读性（权重 15%）

```
function scoreReadability(name):
  各字常用度 = 2.md char.frequency（0~1）
  平均常用度 = mean(frequency(c))
  笔画惩罚 = 任意字笔画 > 25 扣 0.2，> 30 扣 0.5

  得分 = 平均常用度 × 0.7 - 笔画惩罚 + 0.3 × 书写复杂度评分
  书写复杂度评分：左右结构 > 包围结构 > 上下结构 > 独体字

  归一化到 0~1
```

#### 1.2.5 长度结构（权重 10%）

```
function scoreLength(name, type):
  个人姓名：2 字 = 0.9，3 字 = 1.0，4 字 = 0.7，其他 = 0.3
  公司/品牌：2 字 = 0.8，3~4 字 = 1.0，5~6 字 = 0.7，>6 = 0.4
  产品名：2~3 字 = 1.0，4~5 字 = 0.7，其他 = 0.4
```

#### 1.2.6 重复惩罚（权重 10%）

```
function scoreRepeat(name):
  字频统计 = count(chars)

  有重复字：
    2 次重复 → 0.5
    3 次重复 → 0.2
    >3 次重复 → 0

  无重复字：1.0
```

#### 1.2.7 AI 综合评分（权重 10%）

见第二节 AI 增强层方案。

### 1.3 聚合公式

```
totalScore = Σ(weight_i × score_i)

最终输出 = round(totalScore × 100, 1)  // 0~100 分

防作弊下限：若任意基础维度（字义/音律/常用度）< 0.2，总分不高于 30
```

### 1.4 示例计算

```
名字："知远"（个人姓名，风格=现代）

字义：知(0.8) 远(0.85) → 0.825 × 20% = 16.5
音律：zhī(1) yuǎn(3) → 声调变化 2，无重复 → 0.85 × 20% = 17.0
风格："知"=现代/文学 "远"=现代/文学 → 匹配度 1.0 → 0.15 × 15% = 2.25
常用度：知(0.6) 远(0.5) → 0.55 × 15% = 8.25
长度：3 字(含姓) → 1.0 × 10% = 10.0
重复：无 → 1.0 × 10% = 10.0
AI 评分：0.8 × 10% = 8.0

总分 = 16.5 + 17.0 + 2.25 + 8.25 + 10.0 + 10.0 + 8.0 = 72.0
```

## 二、AI 增强层方案

### 2.1 模型选型

| 场景 | 推荐模型 | 替代方案 | 成本预估 |
|------|---------|---------|---------|
| 名字解释（单名） | GPT-4o-mini / Claude 3 Haiku | 本地 Qwen2.5-7B | ~¥0.001/次 |
| 品牌故事/Slogan | GPT-4o / Claude 3.5 Sonnet | DeepSeek-V3 | ~¥0.01/次 |
| 风格匹配/替代名推荐 | GPT-4o-mini | 本地 Qwen2.5-7B | ~¥0.002/次 |
| 批量内容生成（SEO） | GPT-4o-mini | 批处理 + 缓存 | ~¥0.0005/次 |

**推荐策略**：使用 GPT-4o-mini 覆盖 90% 场景，付费用户升级到 GPT-4o 获得品牌故事。

### 2.2 提示词策略

#### 名字解释模板

```
你是一位中文起名专家。请为名字「{name}」（类型：{type}，风格：{style}）
提供一段 50~100 字的中文解释，包括：
1. 每个字的寓意
2. 组合后的整体意境
3. 适合的行业/场景

要求：语言优美、积极正面、不超过 100 字。
```

#### 品牌故事模板

```
请为公司名「{name}」写一段 100~150 字的品牌故事。
行业：{industry}
品牌定位：{positioning}
故事应包括品牌使命和愿景，语言有感染力。
```

### 2.3 回退策略

```
if AI 服务可用:
  使用 AI 输出
else if 命中缓存:
  返回缓存结果
else:
  使用规则生成模板文本（如：「{name}寓意{字义1}与{字义2}的结合，代表{积极义}」）
  标记为「AI 不可用」状态，待恢复后异步补充
```

### 2.4 质量验证

```
离线评估（发布前）：
  - 收集 500 个样本，人工评分 1~5 分
  - 要求平均分 ≥ 4.0，不合格率 < 5%
  - 检查：有无负面联想、事实错误、格式问题

在线监控（上线后）：
  - 用户反馈按钮：「解释有帮助 / 无帮助」
  - 每周抽样 100 条 review
  - 自动检测：包含禁用词、超过字数限制 → 降级到规则模板
```

## 三、性能目标与缓存策略

### 3.1 SLA 指标

| 指标 | 目标 | 严重告警阈值 |
|------|------|-------------|
| 名字生成 P50 延迟 | < 200ms | > 500ms |
| 名字生成 P95 延迟 | < 500ms | > 1s |
| AI 解释 P95 延迟 | < 2s | > 5s |
| 系统可用性 | 99.5% | < 99.0% |
| 并发生成能力 | 100 QPS | 过载保护 |
| 每日生成容量 | 10 万次 | 接近上限告警 |

### 3.2 并发估算

```
初期用户：1000 DAU，每人平均生成 5 次 → 5000 生成/天
峰值系数：3x（活动/推广期）
峰值吞吐：5000 × 3 / (3600 × 6) ≈ 0.7 QPS  →  预留余量到 10 QPS

中期目标（6个月）：10000 DAU → 预留到 100 QPS
```

### 3.3 缓存详细设计

#### 缓存分层

| 层级 | 存储 | 缓存内容 | TTL | 预估内存 |
|------|------|---------|-----|---------|
| L1（进程内） | lru_cache / memory | 字库、拼音、字义 | 热加载（与数据同生命周期） | ~50MB |
| L2（分布式） | Redis | 评分结果、热门名字 | 24h | ~200MB |
| L3（CDN） | CDN/边缘节点 | SEO 页面、静态内容 | 7d | — |

#### Redis 缓存设计

```
Key 模式：
  score:{name}:{type}:{style}     → 评分结果（JSON）
  popular:{type}:{industry}       → 热门名字列表（TTL 1h）
  ai:{name}:{type}:{style}:{lang} → AI 解释结果（TTL 7d）
  rate_limit:{user_id}:{date}     → 调用次数计数（TTL 1d）

淘汰策略：
  全部使用 TTL 自动过期
  内存接近上限时走 allkeys-lru
  热门名字列表用 ZSET，保留 Top 1000

缓存穿透防护：
  对不存在的名字/字，缓存空值（TTL 60s）
  布隆过滤器前置（Bloom Filter）拦截明显无效输入
```

### 3.4 缓存更新策略

```
写操作：
  - 词库更新 → 广播缓存失效事件
  - 管理员手动刷新 → 删除对应 Key

预热策略：
  - 启动时加载核心字库到 L1
  - 定时任务每日刷新热门名字 Top 1000 到 L2
```

## 四、测试与质量保障

### 4.1 数据管线测试

| 测试类型 | 覆盖内容 | 频率 |
|---------|---------|------|
| 完整性检查 | 每个字必有拼音、字义字段 | 每次数据更新 |
| 冲突检测 | 多音字是否标记、负面词是否误入 | 每次数据更新 |
| 格式校验 | JSON Schema 验证、UTF-8 编码 | CI 阶段 |
| 覆盖率检查 | 核心字库 ≥ 5000 字，扩展库 ≥ 20000 字 | 每次发布 |

### 4.2 生成引擎测试

```
# 单元测试
- 输入空值 → 返回错误
- 输入非法字符 → 返回错误
- 2 字/3 字/多字生成 → 返回正确长度
- 指定风格 → 生成结果含指定风格标签
- 排除字列表 → 生成结果不含排除字

# 性能测试
- 单次生成耗时 < 50ms（不包含 AI）
- 100 并发线程下 P95 < 200ms
- 内存泄漏检测：连续生成 10000 次，内存增长 < 10MB

# 回归测试集
- 固定 100 组输入参数，每次发布对比输出数量和质量
- 评分结果浮动 < 5%（因数据更新导致的合理变化需确认）
```

### 4.3 评分系统校准

```
校准流程：
  1. 人工标注 300 个名字（每项维度评分 + 综合评分）
  2. 比较算法输出与人工评分的 Spearman 相关系数
  3. 要求相关系数 ≥ 0.8
  4. 若有偏差，调整权重或评分函数参数
  5. 每季度重新校准一次

A/B 测试（线上）：
  对照组：旧评分算法
  实验组：新评分算法
  指标：用户点击率、生成后转化率、用户满意度反馈
```

### 4.4 AI 输出质量评估

```
# 离线评估集
- 100 个名字 × 3 种类型 = 300 条
- 评估维度：准确性、流畅性、创意性（各 1~5 分）
- 接受标准：平均分 ≥ 4.0，最低分 ≥ 3.0

# 自动化检测
- 长度检查：超出限制则截断
- 敏感词检测：包含禁用词则降级
- 语言检测：输出必须为中文
```

## 五、法规合规

### 5.1 个人姓名相关规定

根据《公安部关于执行〈姓名登记条例〉的若干规定》：

| 要求 | 实现方式 |
|------|---------|
| 姓名长度：汉族 ≥ 2 字 ≤ 6 字 | 生成时校验，超出提示用户 |
| 不得使用已简化的繁体字 | 字库标记「简化字」字段，排除未简化的繁体字 |
| 不得使用异体字（非姓氏） | 比对《通用规范汉字表》，排除三级字表外异体字 |
| 不得使用自造字 | 只从 Unicode CJK + 规范字表选字 |
| 不得使用拼音字母、阿拉伯数字 | 输入过滤 |
| 姓氏随父/母，可选取其他直系长辈血亲姓氏 | 产品提示，不做算法限制 |
| 改名需有正当理由 | 产品文案中提示 |

### 5.2 企业名称相关规定

根据《企业名称登记管理规定》（市场监管总局令第 82 号）：

| 要求 | 实现方式 |
|------|---------|
| 名称结构：行政区划+字号+行业+组织形式 | 生成模板：{字号}+{行业限定词}+{组织形式} |
| 字号 ≥ 2 字 | 生成时校验 |
| 不得使用县级以上行政区划（字号） | 行政区划名称过滤词库 |
| 不得使用行业通用词单独作为字号 | 常见行业词词典（科技、文化、贸易…）处理 |
| 不得使用政党名称、国家机关名称 | 禁用词库 |
| 不得含有误导性内容 | 负面词 + AI 审核 |
| 同行业不得重名 | 需对接国家企业信用信息公示系统（商业版） |

### 5.3 禁限用字清单

| 类型 | 来源 | 处理 |
|------|------|------|
| 政治敏感字 | 《姓名登记条例》附件 | 纳入负面库「严重不适」级别 |
| 歧视性字 | 民政部公告 | 纳入负面库 |
| 不吉利字（亡、丧、狱…） | 传统文化 + 民政部参考 | 纳入负面库 |
| 生僻字（笔画 > 30 或 Unicode 扩展区 B 以外） | 《通用规范汉字表》 | 标记生僻，默认不参与生成 |
| 企业名禁用字（全国性、国际、集团—需审批） | 市场监管总局 | 非特许用户生成时自动标记 |

### 5.4 数据合规

```
- 字库数据：仅使用公开、开源数据（CEDICT 属 CC BY-SA 4.0）
- 姓名语料：Chinese Names Corpus 仅用于统计，不直接展示原始数据
- 用户输入的个人信息（如宝宝姓氏）：不得存储明文超过 30 天
- 数据分析用数据：脱敏后使用，不可还原个人身份
```

## 六、用户隐私框架

### 6.1 数据收集范围

| 数据类型 | 收集目的 | 存储期限 | 是否必需 |
|---------|---------|---------|---------|
| 姓氏 | 起名生成 | 会话结束删除（非会员），30 天（会员） | 是 |
| 生成参数（风格/行业/类型） | 生成 + 分析优化 | 30 天（脱敏） | 是 |
| 生成结果（名字） | 展示 + 优化词库 | 匿名化后长期 | 是 |
| 用户反馈 | 质量优化 | 长期 | 否 |
| 手机号（会员） | 付费 + 身份识别 | 会员存续期 | 付费必需 |
| 微信 OpenID | 小程序身份 | 长期 | 是 |

### 6.2 存储与脱敏

```
# 脱敏规则
- 姓氏+生成参数 → 存储前去除用户标识，仅保留聚合统计
- 用户反馈 → 关联匿名 ID（非手机号/OpenID）
- 数据分析 → 最小聚合单元 ≥ 10 条记录

# 加密要求
- 手机号：AES-256 加密存储
- OpenID：仅用于微信登录，不与其他数据关联
- 数据库备份：整体加密
```

### 6.3 隐私政策要点

```
1. 明确说明收集哪些数据及用途
2. 用户可申请删除所有个人数据
3. 不使用用户数据训练 AI 模型（除非另行授权）
4. 不向第三方出售用户数据
5. 儿童个人信息保护：不满 14 周岁用户需监护人同意
6. 数据跨境：所有数据存储在中国大陆服务器
```

## 七、错误处理与降级策略

### 7.1 错误分类与处理

| 错误类型 | 场景 | 处理方式 | 用户可见表现 |
|---------|------|---------|------------|
| 数据源异常 | CEDICT/字库加载失败 | 使用本地备份文件 + 告警 | 无影响（系统自动恢复） |
| 生成空集 | 所有字被过滤（负面/风格/排除） | 放宽条件（降低风格匹配阈值 → 0.3） | 正常返回（可能风格匹配稍弱） |
| | 放宽后仍为空 | 返回提示：「没有找到符合条件的结果，请调整输入参数」 | 用户可见提示 |
| AI 不可用 | API 超时/限流/错误 | 降级到规则模板 | 名字无 AI 解释，标记「AI 解释暂不可用」 |
| 缓存穿透 | 大量新名字请求未命中缓存 | 回源计算 + 布隆过滤器拦截无效请求 | 无影响（延迟稍高） |
| 评分异常 | 某维度计算溢出/NaN | 该维度取默认值 0.5，记录日志告警 | 无影响 |
| 数据库故障 | Redis/PostgreSQL 宕机 | 降级：只读本地缓存 + 限流 | 功能受限（无法保存收藏等） |
| API 过载 | 并发超过容量 | 限流（429）+ 排队 | 提示「系统繁忙，请稍后重试」 |

### 7.2 重试与熔断

```
# AI 调用
首次失败 → 等待 500ms 重试（最多 2 次）
连续 5 次失败 → 熔断 60s → 降级到规则模板

# 数据库查询
首次失败 → 重试 1 次 → 仍失败则走本地缓存
若本地缓存也无数据 → 返回空（非关键路径）

# 外部数据源
每日定时更新；若拉取失败 → 使用上一日数据 + 告警
连续 3 日失败 → P1 告警通知管理员
```

### 7.3 监控告警

| 指标 | 告警条件 | 通知渠道 |
|------|---------|---------|
| 生成 P95 延迟 > 1s | ≥ 1min | 即时通信（企业微信/钉钉） |
| AI 调用失败率 > 5% | ≥ 5min | 即时通信 |
| 系统可用性 < 99.5% | ≥ 5min | 电话告警（P1） |
| 内存/CPU > 85% | ≥ 10min | 即时通信 |
| 缓存命中率 < 70% | ≥ 30min | 邮件 |

## 八、部署与 DevOps

### 8.1 部署架构

```
用户层
  ├── 微信小程序 → CDN + API Gateway
  ├── H5/WEB → CDN + Nginx
  └── API 客户端 → API Gateway → WAF

网关层
  └── Kong / Nginx → 限流 + 鉴权 + 路由

应用层（容器化，K8s）
  ├── name-generator（无状态，HPA 2~10 pod）
  ├── ai-proxy（AI 调用封装 + 缓存 + 熔断）
  ├── api-server（RESTful，对外接口）
  └── admin-server（后台管理）

数据层
  ├── PostgreSQL / SQLite（词库 + 用户数据）
  ├── Redis（缓存 + 限流计数器）
  └── S3 / OSS（静态资源 + 备份）

可选：Serverless（AI 解释场景，避免常驻资源浪费）
```

### 8.2 CI/CD

```
# CI（每次 Push）
  lint（代码规范）
  typecheck（类型检查）
  unit test（单测覆盖 > 80%）
  build（Docker 镜像构建）
  data validation（词库完整性校验）

# CD（合并到 main）
  staging 部署 → integration test
  staging 通过后 → 手动确认 → production 部署
  回滚：保留前 5 个版本，一键回滚

# 发布频率
  功能发布：每周 1 次（周四）
  紧急修复：按需（建立 hotfix 分支）
  词库更新：每月 1 次
```

## 九、用户研究与评分验证

### 9.1 评分权重验证

```
# 方法一：Conjoint Analysis（联合分析）
招募 100 名目标用户，展示 2 组名字（仅改变某个维度）
统计用户偏好 → 反推各维度的实际权重

# 方法二：A/B 测试
对照组：当前权重
实验组：调整后的权重
关键指标：用户对 Top 3 名字的点击率、生成后分享率
有效样本量：每组 ≥ 1000 次生成
运行周期：2 周

# 方法三：专家评审
邀请 3~5 名起名行业专家（姓名学从业者/品牌顾问）
对算法推荐的 Top 5 名字打分 1~5
频率：每季度一次
```

### 9.2 用户满意度跟踪

```
# 即时反馈（每次生成后）
「您对这个名字满意吗？」→ 非常满意 / 一般 / 不满意
目标：满意率 ≥ 70%

# 结果跟踪（生成后 7 天）
如果用户选择了某个名字，询问：
「您最终选用了哪个名字？」→ 记录已选名字 + 评分
目标：了解算法高分与用户选择的匹配度

# NPS 调查（每月一次）
「您有多大可能向朋友推荐我们的起名服务？」(0~10)
目标：NPS ≥ 30
```

### 9.3 持续优化

```
每月分析：
  1. 评分分布：是否正态？（若不是，校准评分函数）
  2. 用户选择 vs 算法 Top 1 的吻合度：目标 > 60%
  3. 付费用户生成参数分布：发现高价值场景
  4. AI 输出使用率 + 正反馈率

季度迭代：
  1. 根据 Conjoint Analysis 更新权重
  2. 扩展/精简风格标签
  3. 优化负面词库（移除误杀，增加新发现的负面词）
```

## 十、数据规模与增长规划

### 10.1 存储估算

| 数据类型 | 记录数 | 单条大小 | 总大小（内存） | 总大小（磁盘） |
|---------|-------|---------|-------------|-------------|
| 核心字库 | 8,000 | ~200B | 1.6MB | 1.6MB |
| 扩展字库 | 50,000 | ~200B | 10MB | 10MB |
| 行业/风格字库 | 30,000 | ~200B | 6MB | 6MB |
| 负面过滤词 | 10,000 | ~100B | 1MB | 1MB |
| 组合词表 | 50,000 | ~300B | 15MB | 15MB |
| 拼音索引 | 8,000 | ~100B | 0.8MB | 0.8MB |
| Redis 缓存 | — | — | ~200MB | — |
| PostgreSQL（含用户数据） | — | — | — | ~1GB（初期） |
| **合计** | | | **~234MB** | **~1.03GB** |

### 10.2 增长模型

```
# 字库增长
核心字库：基本不变（8,000 封顶）
扩展字库：每半年 + 5,000 → 2 年内达 70,000
行业词：随行业扩展，每月 + 500 → 2 年内达 80,000

# 用户数据增长
初期（1~3 月）：1000 DAU → 用户数据 ~100MB
中期（4~12 月）：10000 DAU → 用户数据 ~1GB
成熟期（2 年+）：50000 DAU → 用户数据 ~10GB

# 缓存增长
随热门名字数量增长，80% 请求集中在 5000 个名字以内
Redis 内存 200MB 可满足 2 年需求 → 届时评估扩容
```

### 10.3 词库更新流程

```
1. 新增词收集：AI 辅助 + 人工审核
2. 更新词库文件 → 提交 Git PR
3. CI 验证完整性 + 冲突检测
4. 合并到 main → 自动构建新版词库包
5. 灰度发布（10% 流量用新词库，观察 24h）
6. 全量发布
7. 旧版词库数据保留 30 天用于回滚

频率：每月一次常规更新，紧急修复随时发布
```

### 10.4 容量规划

```
# 数据库
单表不超过 5000 万行（PostgreSQL 分区能力）
达到阈值前做好分表/分区规划

# 应用服务器
每台 2C4G 可支撑 50 QPS（不包含 AI）
K8s HPA 配置：CPU > 70% 或 QPS > 50 时扩容

# 扩容时间线
0~6 月：1~2 台服务器，单节点部署
6~18 月：4~8 台，K8s 集群
18 月+：10~20 台，多区域部署
```
